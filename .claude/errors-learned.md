## [2026-07-21] — Netlify DB no inyecta connection string en functions "Lambda compatibility"

**Contexto:** Fase 1, implementando `netlify/functions/brands.ts` (CRUD) usando `@netlify/database` (`getDatabase()`).
**Error:** `MissingDatabaseConnectionError: The environment has not been configured to use Netlify Database. You must supply the connectionString option when calling getDatabase()`. Ocurría en producción aunque el deploy mostraba "Migrations applied" y el panel Database confirmaba "Your database is ready!".
**Causa raíz:** La función estaba escrita en el formato clásico `export const handler: Handler = async (event) => {...}` (estilo Lambda/`@netlify/functions` v1). Netlify solo inyecta automáticamente la connection string de Netlify DB en el formato moderno de functions (`export default async (req: Request) => {...}`, basado en Web Request/Response). El formato clásico corre en "modo de compatibilidad Lambda" y no recibe esa inyección automática.
**Fix aplicado:** Se reescribieron `brands.ts` y `generate-batch.ts` al formato v2 (`export default async (req: Request) => new Response(...)`), usando `new URL(req.url).searchParams` en vez de `event.queryStringParameters` y `await req.json()` en vez de `JSON.parse(event.body)`. Tras el redeploy, `getDatabase()` funcionó sin pasar `connectionString` manualmente.
**Prevención:** En este proyecto, TODAS las Netlify Functions que usen `@netlify/database` deben escribirse en formato v2 (`export default`), nunca en formato clásico `Handler`. `health.ts` (que no toca la DB) se dejó en formato clásico porque funciona igual para ese caso simple, pero cualquier function nueva que use la DB debe partir del patrón v2 desde el inicio.
**Archivos:** `netlify/functions/brands.ts`, `netlify/functions/generate-batch.ts`

## [2026-07-22] — Auditoría completa de Fases 1-4: 9 hallazgos, incluyendo API pública sin protección

**Contexto:** Auditoría a pedido del usuario sobre las 4 fases construidas hasta ahora, antes de conectar las API keys reales.
**Error/hallazgos:** 9 problemas reales verificados (varios probados en vivo contra el sitio desplegado): (1) cero autenticación en toda la API/dashboard — violaba el plan explícitamente; (2) `image_urls` guardaba ruta relativa, rota para que Postiz la descargue; (3) `max_tokens` insuficiente para lotes grandes en `generate-batch.ts`; (4) `post_metrics` es una tabla de snapshots (una fila nueva por sync) pero 3 functions la sumaban sin dedup, sobre-contando métricas; (5) `regenerate-copy.ts` no forzaba re-aprobación tras editar el texto; (6) estado local `copyText` en el Kanban no se resincronizaba tras regenerar; (7) casi ninguna function tenía `try/catch`, a diferencia de `brands.ts`; (8) el matching de métricas de Metricool ignoraba la red social del post; (9) `posts.ts` permitía saltar directo a `published`/`scheduled` vía PATCH manual.
**Causa raíz:** Construcción rápida multi-fase en una sola sesión sin pase de revisión intermedio; cada function se escribió aislada sin comparar consistencia contra las anteriores (ej. brands.ts sí tenía try/catch, las siguientes no lo replicaron).
**Fix aplicado:** Los 8 hallazgos de código se corrigieron directo (ver commits `41ca983` y el commit de autenticación). Para el #1, se intentó primero la protección de sitio nativa de Netlify (`update-visitor-access-controls` vía MCP) pero es una función de pago (plan Team Dev no la incluye, error 422 + "Upgrade to unlock" en el dashboard) — se implementó en su lugar autenticación propia: `POST /api/login` con contraseña (bcrypt) devuelve un token HMAC (`netlify/functions/lib/auth.ts`), y todas las functions sensibles llaman `isAuthorized(req)`.
**Prevención:** (a) Antes de asumir que una función de plataforma (ej. password protection de Netlify) está disponible, verificar el plan/tier real, no solo la documentación general. (b) Cuando se agregue una function nueva a este proyecto, replicar el patrón de `try/catch` + `isAuthorized(req)` desde el primer commit, no como parche posterior. (c) Tablas de series de tiempo (con columna `captured_at`/similar) requieren `DISTINCT ON` o subquery de "último snapshot" en CUALQUIER agregación — nunca un JOIN simple.
**Archivos:** `netlify/functions/generate-image.ts`, `generate-batch.ts`, `metrics.ts`, `weekly-report.ts`, `analyze-brand.ts`, `regenerate-copy.ts`, `schedule-post.ts`, `postiz-webhook.ts`, `sync-metrics.ts`, `posts.ts`, `lib/auth.ts`, `login.ts`, `health.ts`, `brands.ts`, `src/pages/Pipeline.tsx`, `src/lib/api.ts`, `src/lib/auth.ts`, `src/App.tsx`

## [2026-07-22] — Dos "free tier" resultaron no ser viables: Oracle Always Free (capacidad) y Metricool (API de pago)

**Contexto:** Fase 3 (Postiz en VPS) y Fase 4 (métricas de Metricool), siguiendo el requisito explícito del usuario de mantener todo el sistema gratuito salvo Claude/Gemini.
**Error/hallazgos:**
1. Oracle Cloud "Always Free" (VM.Standard.A1.Flex, la que da 4 OCPU/24GB gratis de por vida) devolvió `Error de API: Capacidad insuficiente en la unidad VM.Standard.A1.Flex en el dominio de disponibilidad AD-1` en 4 intentos reales de creación, en la región Mexico Northeast (Monterrey). Además, la cuenta del tenant quedó limitada a 1 sola región (`Ha superado el número máximo de regiones permitidas para el arrendamiento` al intentar suscribir otra región) — no se pudo ni siquiera cambiar de región dentro de la misma cuenta.
2. Metricool: la pestaña "API" en Ajustes de cuenta mostró "Acceso API — Planes Advanced y Custom" con botón "Actualiza tu plan" — el plan gratis (el que se pensaba usar) no incluye token de API, solo el dashboard web.
**Causa raíz:** Ambos son casos de documentación/expectativa general ("tiene free tier") que no se verificó contra la disponibilidad/alcance REAL del free tier antes de comprometerse con la arquitectura. La capacidad de Oracle Always Free es notoriamente volátil por región (hecho conocido de la comunidad, no específico de esta cuenta). El acceso a API de Metricool nunca estuvo incluido en su plan gratis (solo exportar/dashboard).
**Fix aplicado:** Se abandonó Postiz/VPS por completo — Fase 3 se reconstruyó como integración directa con la Graph API de Meta desde las propias Netlify Functions (`netlify/functions/lib/meta.ts`, `publish-due-posts.ts` como scheduled function porque Instagram no soporta programación nativa vía API). Cero servidor adicional, decisión del usuario tras "me rindo con Oracle, vamos a usar alguna otra alternativa gratuita" → prefirió preguntar "¿por qué necesitamos esto? ¿no puede vivir en Netlify?" en vez de perseguir otro proveedor de VPS. Metricool quedó pendiente sin resolver — el código de Fase 4 existe pero inactivo hasta que el usuario decida pagar el plan Advanced o buscar otra fuente de métricas.
**Prevención:** (a) Antes de diseñar una arquitectura alrededor de un "free tier", verificar en la consola real de esa cuenta qué incluye exactamente el plan gratis (no asumir desde memoria/documentación general) — esto aplica doble a límites de capacidad regional (Oracle) y a gating de features por plan (Metricool). (b) Cuando una pieza de infraestructura de terceros se vuelve un bloqueante recurrente, es válido y a veces mejor cuestionar si esa pieza es necesaria en absoluto (ver Opción A: integración directa) en vez de seguir intentando distintos proveedores para la misma pieza.
**Archivos:** `netlify/functions/lib/meta.ts`, `schedule-post.ts`, `publish-due-posts.ts`, `brands.ts`, `netlify/database/migrations/20260722010000_add_meta_fields/migration.sql` (eliminado: `postiz-webhook.ts`)
